Diretrizes de Desenvolvimento em Modo Stub

Integração com Autoridades Fiscais — Estratégia para Ambientes sem Credenciais Reais

Versão 2.0 • Maio 2026

Classificação: Interno — Uso restrito à equipe técnica

1. Contexto e Objetivo

O projeto ELLAN LAB possui integração com múltiplas autoridades fiscais globais para emissão, autorização, cancelamento e correção de documentos eletrônicos. Enquanto as credenciais de homologação real não estiverem disponíveis — seja por indisponibilidade de contato com os órgãos competentes ou por falta de cadastro formal — a equipe deve avançar o desenvolvimento usando uma camada de simulação controlada: o Modo Stub.

Este documento define padrões, contratos de interface, estrutura de dados e boas práticas para que o desenvolvimento em stub seja diretamente substituível pela integração real sem necessidade de refatoração de alto impacto. A versão 2.0 expande a cobertura para Estados Unidos, Austrália, Tailândia, Alemanha, Polônia, Canadá e França, incluindo regionalidades como Califórnia, Texas e Quebec.

⚠️ Premissa Fundamental

Todo código escrito em modo stub DEVE ser estruturado de forma que a substituição pelo provedor real exija apenas a troca do adaptador de infraestrutura, sem alterar regras de negócio ou contratos de domínio. O stub é um adapter de desenvolvimento — não uma solução definitiva.

2. Arquitetura da Camada Stub

2.1 Padrão de Separação por Interface

A equipe deve adotar o padrão Port & Adapter (Hexagonal Architecture). Cada autoridade fiscal é representada por uma interface (porta), e o stub é um adaptador concreto dessa porta. A troca para o adaptador real acontece apenas na camada de infraestrutura.

// Exemplo em TypeScript
// PORTA — nunca muda, é o contrato de domínio
interface IFiscalProvider {
  authorize(doc: FiscalDocument): Promise<AuthorizeResult>;
  cancel(doc: FiscalDocument, reason: string): Promise<CancelResult>;
  correct(doc: FiscalDocument, correction: CorrectionData): Promise<CorrectResult>;
  status(key: string): Promise<StatusResult>;
}

// ADAPTADOR STUB — substitui o real durante desenvolvimento
class FiscalProviderStub implements IFiscalProvider {
  async authorize(doc) { return StubFactory.authorizeSuccess(doc); }
  async cancel(doc, reason) { return StubFactory.cancelSuccess(doc); }
  async correct(doc, correction) { return StubFactory.correctSuccess(doc); }
  async status(key) { return StubFactory.statusApproved(key); }
}

7. Regras de Negócio Específicas (Stub Obrigatório)

7.1 Estados Unidos (IRS e Estados)

  • IRS MeF: Gerar SubmissionId com formato IRS-TAX-YEAR-XXXX, aceitar códigos de rejeição como R0000-502 (Business Rule Violation).
  • Califórnia (CDTFA): Simular registro de Seller’s Permit, campo "CDTFA-XXXX" e validação de periodicidade de filing (mensal/trimestral).
  • Texas (Comptroller): Simular Sales Tax Permit com validação de Out-of-State dealers, endpoint /webfile deve retornar Webfile ID.
  • Regionalidade: diferenças entre estados — no stub é possível alternar via header X-State-Code.

7.2 Austrália (ATO)

  • e-Invoicing via Peppol: validar ABN (Australian Business Number) com algoritmo de checksum.
  • BAS (Business Activity Statement): simular campos G1-G11 com validação de GST, respostas via Standard Business Reporting (SBR).

7.3 Tailândia (RD)

  • e-Tax Invoice: formato XML segundo PDPA laws, número de identificação fiscal (TIN) deve ter 13 dígitos (formato tailandês). Simular aprovação de Withholding Tax Certificate (Por Por 30).

7.4 Alemanha (XRechnung / Peppol)

  • Presença obrigatória do Leitweg-ID (município) e Buyer Reference. Stub deve retornar acknowledgment Peppol AS4 com MessageID único.

7.5 Polônia (KSeF - Krajowy System e-Faktur)

  • Simular sessão autenticada por token (SessionToken). Invoices devem possuir número de referência (Numer referencyjny) gerado no stub com prefixo KSeF/YYYY/MM.
  • JPK_VAT: validar estrutura XML e rejeitar se falta pola 6/7.

7.6 Canadá (CRA + Revenu Québec)

  • CRA: GST/HST NETFILE - simular AccessCode e Business Number (9 dígitos + RP/RC).
  • Quebec: QST e RL-1 slip com validação de NEQ (Número d'entreprise du Québec). Stub deve aceitar formulário CO-17 para imposto de renda corporativo Quebec.

7.7 França (DGFiP / Chorus Pro)

  • Factur-X / Chorus Pro: simular identificador Chorus (SIRET + indice de emission). O stub deve reproduzir a mensagem "Statut: Acceptée par le fournisseur".

10. Boas Práticas e Regras da Equipe

10.1 O que SEMPRE fazer (Global)

  • Manter compatibilidade com múltiplos formatos regionais (XML/JSON/UBN/Peppol).
  • Simular latências realistas de cada autoridade: ≥200ms para CRA, ≤100ms para IRS sandbox.
  • Versionar fixtures com tag da especificação fiscal (ex: CFDIv4, XRechnung1.2).

10.2 O que NUNCA fazer

  • Retornar envelope universal sem campo "provider" específico.
  • Utilizar dados reais de certificados de produção nos fixtures de stub.
  • Ignorar validação de regras regionais (ex: não validar formato de TIN australiano pode quebrar contrato).
✅ Definição de "Stub Global Pronto para Migração"

1. Todos os endpoints regionais documentados na seção 3 respondem com envelope padronizado.
2. Fixtures atualizados conforme última especificação técnica (2025/2026).
3. Testes unitários com cobertura de cenários de erro regionais.
4. Troca via FISCAL_MODE=real deve apontar para adaptador autenticado, sem alterar contrato.
