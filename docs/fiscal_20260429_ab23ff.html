Diretrizes de Desenvolvimento em Modo Stub

Integração com Autoridades Fiscais — Estratégia para Ambientes sem Credenciais Reais

Versão 1.0 • Abril 2025

Classificação: Interno — Uso restrito à equipe técnica

1. Contexto e Objetivo

O projeto ELLAN LAB possui integração com múltiplas autoridades fiscais globais para emissão, autorização, cancelamento e correção de documentos eletrônicos. Enquanto as credenciais de homologação real não estiverem disponíveis — seja por indisponibilidade de contato com os órgãos competentes ou por falta de cadastro formal — a equipe deve avançar o desenvolvimento usando uma camada de simulação controlada: o Modo Stub.

Este documento define padrões, contratos de interface, estrutura de dados e boas práticas para que o desenvolvimento em stub seja diretamente substituível pela integração real sem necessidade de refatoração de alto impacto.

⚠️ Premissa Fundamental

Todo código escrito em modo stub DEVE ser estruturado de forma que a substituição pelo provedor real exija apenas a troca do adaptador de infraestrutura, sem alterar regras de negócio ou contratos de domínio. O stub é um adapter de desenvolvimento — não uma solução definitiva.

2. Arquitetura da Camada Stub

2.1 Padrão de Separação por Interface

A equipe deve adotar o padrão Port & Adapter (Hexagonal Architecture). Cada autoridade fiscal é representada por uma interface (porta), e o stub é um adaptador concreto dessa porta. A troca para o adaptador real acontece apenas na camada de infraestrutura.

// Exemplo em TypeScript
// PORTA — nunca muda, é o contrato de domínio
interface IFiscalProvider {
  authorize(doc: FiscalDocument): Promise<AuthorizeResult>;
  cancel(doc: FiscalDocument, reason: string): Promise<CancelResult>;
  correct(doc: FiscalDocument, correction: CorrectionData): Promise<CorrectResult>;
  status(key: string): Promise<StatusResult>;
}

// ADAPTADOR STUB — substitui o real durante desenvolvimento
class FiscalProviderStub implements IFiscalProvider {
  async authorize(doc) { return StubFactory.authorizeSuccess(doc); }
  async cancel(doc, reason) { return StubFactory.cancelSuccess(doc); }
  async correct(doc, correction) { return StubFactory.correctSuccess(doc); }
  async status(key) { return StubFactory.statusApproved(key); }
}

// REGISTRO — troca via variável de ambiente
const provider = process.env.FISCAL_MODE === 'real'
  ? new SefazNFeProvider(credentials)
  : new FiscalProviderStub();

4. Contratos de Resposta do Stub

4.1 Envelope de Resposta Universal

Todas as respostas do stub, independente da autoridade fiscal, devem seguir um envelope único para facilitar o parsing na camada de aplicação:

// StubResult — envelope universal
{
  "stub": true,                        // Flag obrigatória — identifica resposta simulada
  "provider": "SEFAZ_SP",              // Identificador do provedor simulado
  "scenario": "AUTHORIZE_SUCCESS",     // Cenário executado
  "timestamp": "2025-04-29T10:00:00Z",// Timestamp ISO 8601
  "latencyMs": 320,                    // Latência simulada (realismo)
  "payload": {                         // Payload específico do documento
    "protocol": "135251234567890",
    "status": "100",
    "statusDescription": "Autorizado o uso da NF-e",
    "xml": "<nfeProc>...</nfeProc>",   // XML conforme layout real
    "qrCode": "https://www.sefaz.../...",
    "chaveAcesso": "35250429..."
  },
  "rawResponse": "<?xml version=..."   // String bruta simulando o protocolo real
}

5. Implementação da StubFactory

5.1 Modelo de Fábrica com Cenários Configuráveis

// StubFactory.ts
export class StubFactory {
  private static async delay(ms = 300) {
    return new Promise(resolve => setTimeout(resolve, ms));
  }

  static async authorizeSuccess(doc: FiscalDocument): Promise<AuthorizeResult> {
    await this.delay(StubConfig.latencyMs);
    return {
      stub: true,
      scenario: 'AUTHORIZE_SUCCESS',
      protocol: this.generateProtocol(),
      status: '100',
      statusDescription: 'Autorizado o uso da NF-e',
      chaveAcesso: doc.chaveAcesso,
      xml: StubFixtures.load('nfe_authorized.xml'),
      timestamp: new Date().toISOString(),
    };
  }

  static generateProtocol(): string {
    const uf = '35'; // SP
    const aamm = new Date().toISOString().slice(2,4) + new Date().toISOString().slice(5,7);
    const seq = String(Date.now()).slice(-12);
    return `${uf}${aamm}${seq}`;
  }
}

7. Regras de Negócio que o Stub DEVE Respeitar

7.1 Brasil — SEFAZ / SVRS

  • Chave de acesso deve ter 44 dígitos com dígito verificador calculado corretamente (Mod 11)
  • Protocolo de autorização gerado com formato real: cUF(2) + AAMM(4) + sequencial(12)
  • Prazo de cancelamento: stub deve rejeitar cancelamentos com mais de 24 horas da emissão (cStat=301)
  • Limite de 20 cartas de correção por NF-e — stub deve rastrear e rejeitar na 21ª

7.2 Portugal — AT

  • ATCUD (Código Único de Documento) deve ser gerado com hash SHA-1 da chave privada simulada
  • QR Code seguir formato exato dos Despachos 412/2021

7.3 México — SAT / CFDI 4.0

  • UUID do TimbreFiscalDigital deve ser UUID v4 válido gerado pelo 'PAC' stub
  • Versão CFDI: usar 4.0 exclusivamente

10. Boas Práticas e Regras da Equipe

10.1 O que SEMPRE fazer

  • Implementar a interface completa — nunca deixar métodos não implementados
  • Versionar os fixtures no repositório — eles são parte do contrato
  • Testar todos os cenários de erro — stub deve cobrir rejeições, timeouts

10.2 O que NUNCA fazer

  • Usar o stub em produção — flag stub: true deve alertar
  • Retornar estruturas de dados diferentes do adaptador real
  • Hardcodar cenários sem possibilidade de override
✅ Definição de 'Stub Pronto para Migração'

1. Todos os cenários obrigatórios da Seção 4.2 implementados.
2. Fixtures seguem layout oficial da autoridade.
3. IFiscalProvider completamente satisfeita.
4. Testes unitários passam com stub.
5. FISCAL_MODE=real resulta apenas em troca de adaptador.
