================================================================================
ELLAN LAB — Acompanhamento de Sprints · Fiscal & Invoices
================================================================================
Fonte de planejamento: docs/Sprint_Backlog_Fiscal_and_Invoices.pdf
Serviço principal: billing_fiscal_service
Última atualização do plano (PDF): 2026-04-23

Objetivo
--------
Evoluir o billing_fiscal_service do estado stub atual para padrão de indústria de
lockers: NFC-e modelo 65, cálculo de tributos, integrações SEFAZ (BR) e AT (PT),
cancelamento, reconciliação fiscal e integração ao domínio de Orders (contrato
enriquecido + stub coordenado).

Resumo quantitativo
-------------------
  3 sprints eixo Fiscal   (F-1, F-2, F-3)
  3 sprints eixo Invoice  (I-1, I-2, I-3)
  1 sprint Orders (stub)  (O-1)
  ~103 story points total (referência do backlog)

Sequenciamento oficial (dependências)
-------------------------------------
  O-1  →  desbloqueador da cadeia (contrato rico Orders ↔ Fiscal)

  Após O-1, em paralelo:
    F-1  (schema / documento unificado — pré-requisito dos demais fiscais)
    I-1  (integração Orders → invoice completa)

  Depois de F-1 e I-1, em paralelo:
    F-2  (tax engine BR + PT)
    I-2  (cancelamento + entrega DANFE)

  Encerramento produção:
    F-3  (provedores reais NFC-e / SAT / AT — exige F-2 para XML correto)
    I-3  (reconciliação, monitoramento, exports SPED / SAF-T)

Legenda de status (preencher ao longo dos sprints)
----------------------------------------------------
  [ ] Não iniciado   [~] Em andamento   [x] Concluído   [!] Bloqueado / risco

SPRINT F-1 — Schema evolution & document unification (Fiscal)
-------------------------------------------------------------------------------
Duração referência: 2 semanas  |  Esforço: 13 pts
Objetivo: unificar fiscal_documents e invoices; campos obrigatórios NFC-e 65;
          contexto locker no documento. Pré-requisito para os demais sprints
          fiscais.

  User stories
  [x] SCHEMA: fiscal_doc_subtype — valores VARCHAR (NFC_E_65, NFE_55, …) +
        default NFC_E_65 no billing_fiscal_service.invoices
  [x] SCHEMA: emission_mode — ONLINE (+ constantes CONTINGENCY_SVRS, OFFLINE_SAT)
  [x] SCHEMA: locker context na invoice — locker_id, totem_id, slot_label;
        locker_address JSONB
  [x] SCHEMA: emitter & consumer — emitter_*, consumer_*, items_json JSONB
  [x] REFACTOR: leituras passam a preferir invoice no billing_fiscal_service
        (BILLING_FISCAL_SERVICE_URL); fiscal_documents permanece para stub
        local / reimpressão até remoção futura.

  DDL resumo (invoices — referência backlog)
  - locker_id, totem_id, fiscal_doc_subtype, emission_mode
  - emitter_cnpj, emitter_name, consumer_cpf, consumer_name, locker_address
  - items_json (espelho order_items)

  Registro de evolução
  - 2026-04-23: F-1 no billing_fiscal_service — colunas na tabela invoices
    (migrations REQUIRED_COLUMNS + índices), modelo SQLAlchemy, preenchimento
    via invoice_snapshot_fiscal.fiscal_columns_from_order_snapshot no
    ensure_invoice_for_order e generate_invoice; API InvoiceResponse; stub
    SEFAZ-SP usa colunas F-1. Testes: tests/test_invoice_snapshot_fiscal.py
    (pytest na .venv do serviço).
  - 2026-04-23: Redirecionamento de leitura no order_pickup_service —
    integrations/billing_fiscal_client, fiscal_read_adapter.FiscalReadView,
    fiscal_resolve.resolve_fiscal_for_order; internal/status
    (fiscal_invoice_in_billing), public_orders, kiosk identify,
    get_fiscal_document_by_order; /public/fiscal/by-code permanece só
    fiscal_documents (comentário na rota). InvoiceResponse + issued_at etc.
    no billing. Testes: tests/test_fiscal_read_adapter.py.

  [x] API: POST /internal/invoices/{id}/cce-request — fila CORRECTION_REQUESTED → worker CC-e stub (BR/PT/ES)
  [x] API: fluxo reissue — POST /internal/invoices/{id}/reissue preserva `previous_receipt_code`

  SCHEMA: novos InvoiceStatus — CANCELLING, CORRECTION_REQUESTED, COMPLEMENTARY_ISSUED +

    POST /internal/invoices/{id}/cancel-request; worker processa CANCELLING; domain order.refunded;

  - 2026-04-23: I-2 — e-mail (invoice_email_service + delivery log); worker CC-e stub
